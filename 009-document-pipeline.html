// 009 — Document Processing Pipeline · Claude Engineering

DOCUMENT,
PROCESSED.

A five-stage MCP pipeline for claims, invoices, and contracts. Extract structured JSON, mask PII, validate against schema and business rules, route exceptions to a human queue, report token cost. Each stage is its own tool; the model composes them.

build.2026.05 v0.1.0 5 stage tools · MCP SDK 1.x · ajv validation [ ACTIVE ]
// The Brief

Document processing pipelines look simple on a slide. Extract. Validate. Mask. Queue exceptions. Ship. In practice they're a black-box workflow with the model in the middle, doing everything from OCR to business rules to deciding what's PII. Hard to debug, hard to cost, hard to swap a stage out.

This pipeline pulls those concerns apart. Five MCP tools, one per stage. The model is the orchestrator, not the pipeline. Each tool owns its piece of the contract — doc_extract returns a schema and a set of rules, pii_mask redacts by path and by pattern, doc_validate runs schema + business rules and produces an exceptions array, hitl_route turns exceptions into a triaged queue, cost_report totals the tokens.

The interesting design choice: doc_extract doesn't call a model. It returns a work order — task, schema, rules — for the calling agent to execute. No Anthropic API key needed for the demo. Production swaps the work order for an API call without changing any other stage. The pattern survives the boundary.

// 01 — The pipeline 5 stages · 3 patterns · ~1240 LOC source

Five tools, one orchestrator.

Each stage is a separate MCP tool, swappable in isolation. The orchestrator is the model. Three of the five tools are mutations (extract, mask, route); one is critique (validate); one is aggregation (cost). The pipeline is composition.

// stage 1 doc_extract // mutation · work-order Returns task + schema + 13 rules. The model executes against the doc text.
// stage 2 pii_mask // mutation · two-layer Path-based + regex. Free-text fields routed back for contextual review.
// stage 3 doc_validate // critique · redaction-aware ajv schema + 12 business rules. Format checks on redacted fields skip cleanly.
// stage 4 hitl_route // mutation · triage Exceptions → queue. Auto-priority on rule type. Suggested resolutions attached.
// stage 5 cost_report // aggregation Per-stage tokens × rates. Totals, plus per-1k and per-100k scaling.
// Design rationale Separation by stage is separation by concern. Each tool's input contract is the previous tool's output contract — pure functions of the data, no shared state. The pipeline becomes inspectable: drop a tool's trace into the orchestrator, swap it for another, A/B the prompts. The work-order pattern keeps the demo runnable on a Max subscription; production with API-direct execution drops in at doc_extract without touching the four downstream tools.

// 03 — Inside the work-order pattern no API key required · production-portable

Tools return contracts, not completions.

The interesting MCP design choice. doc_extract is a mutation tool that doesn't call a model. It returns a structured work order for the calling agent to execute. Production with API-direct execution drops in without touching any downstream stage.

tools/call · doc_extract [ MUTATION · WORK ORDER ] 
// request
{
  "doc_text": "NORTHSTAR MUTUAL — AUTO CLAIM SUBMISSION\nClaim ID: NSM-2026-0451829\n...",
  "doc_type": "claim"
}

// response
{
  "task": "Extract structured data from this claim document. Conform exactly to the schema. Return JSON only.",
  "doc_type": "claim",
  "schema": { /* full JSON Schema for AutoClaim — types, requireds, patterns, formats */ },
  "extraction_rules": [
    "Output JSON only, conforming exactly to the provided schema. No commentary, no markdown fences.",
    "Use ISO 8601 for all dates (YYYY-MM-DD) and date-times (YYYY-MM-DDTHH:mm:ssZ or with offset).",
    "Currency values: strip $ and commas, return as numbers.",
    "Missing fields → null. Do not invent values.",
    "policyholder.address.state should be the two-letter USPS code.",
    "vehicle.year is an integer, not a string.",
    /* 7 more rules */
  ],
  "output_format": "Plain JSON, valid against the schema. No prose, no fences, no comments.",
  "on_completion": "Pass the extracted JSON to pii_mask with doc_type='claim'.",
  "estimate": { "stage": "extract", "input_tokens": 1955 }
}
// Why the tool doesn't call the model In MCP, the orchestrating agent is already an LLM. Asking the server to spin up another LLM to do extraction is double inference for no gain — and locks every demo to an API key. Returning a work order keeps the model as the executor. The server's job is to encode the contract: which schema, which rules, what to do with the result. The model reads the contract, generates the JSON, hands off to the next stage. Production swaps the work order for an Anthropic API call at this single seam; the downstream tools don't notice.
// In review
// Craft decisions
  • Five tools, one per stage. doc_extract / pii_mask / doc_validate / hitl_route / cost_report. Each owns one contract; the pipeline is composition.
  • The work-order pattern. doc_extract returns a schema + rules + task instead of calling a model. The orchestrator executes. No API key for the demo; the same seam takes a real API call in production.
  • PII masking is two-layer: deterministic regex + path-based redaction first, then a list of free-text fields flagged for the orchestrator to scan contextually. The tool draws the line between what can be caught with rules and what can't.
  • doc_validate is redaction-aware. Format and pattern errors on values matching /^\[.*-REDACTED\]$/ are filtered out. Rules that need PII (policyholder_age against a masked DOB) skip with a note that they should run before mask in production.
  • Business rules live alongside the schema (businessRules array in the JSON Schema file). One file per doc type. Evaluator functions in the validator file. Adding a rule is two changes: schema entry plus evaluator.
  • The orchestrator script (scripts/run-pipeline.mjs) composes all five tools end-to-end and emits a single trace per fixture. Those traces — checked into the repo — are the source-of-truth for the playback on this page.
  • Exceptions get auto-prioritized in hitl_route based on rule type. Missing required fields are high; business rule failures with monetary impact are high; other schema issues are medium; everything else is low.
// What I'd do differently
  • PDF / OCR is not in the loop. Fixtures are plain text. A real version takes a PDF stream, OCRs to text, then enters at doc_extract. The OCR tool is the obvious next addition — and a clean MCP shape: bytes in, text + per-page confidence out.
  • The HITL queue is in-memory. Production persists each entry to a ticket system (Notion, Linear, Jira) and routes by team. The shape — queue_id, priority, suggested_resolution — is portable. The persistence layer isn't.
  • The contextual review pass for pii_mask is documented but not auto-run. The tool returns a list of free-text fields to scan; the orchestrator can call the model on those. A future tool — pii_review — would close that loop.
  • Token cost is estimated via chars / 4. Real API runs return measured token counts; cost_report already accepts a method: 'measured' flag and reports mode accordingly. Demo runs in estimated; production flips it.
  • The schema-with-redactions interaction is handled by post-hoc error filtering. A cleaner version emits a "masked schema" variant per doc type — same shape, PII fields permissive. The current approach is shorter; the cleaner version is more honest.
  • Auto-numbering, line-of-fire test cases, prompt-iteration logging. A second pass would build a proper evals harness around doc_extract — the same harness Project 010 (voice_check v2) is going to need.
