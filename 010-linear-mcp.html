Linear MCP

Linear is an app that software teams use to track their work. I built a tool that lets an AI assistant like Claude help run it. The AI can sort your notifications, plan out a new task, or write the team's weekly update. I also did the careful work most people skip when they build something like this: testing it properly, reviewing how it could be attacked, and checking what it costs to run.

6 tools MCP server TypeScript Open source
The Brief

Linear is where a lot of software teams keep their work. The bug reports, the tasks, who is doing what. An AI assistant cannot see into Linear on its own, so it needs a bridge built for it. That bridge is what this project is.

The easy way to build that bridge is to copy every button and menu Linear has, and hand all of them to the AI. It works, but it leaves the AI to figure everything out, and it does not match how a real person uses Linear. I built it the other way. I looked at what a team lead actually does in Linear each week, picked six of those jobs, and made one tool for each. You ask in plain language, and the AI runs the right tool.

One choice is worth calling out. None of the six tools can change or delete anything in Linear. They only read information and explain it.

The tools were about half the project. The rest was making it something a company could trust. I built a test suite that checks the tools really work. A written security review covers how someone might misuse it and what stops them. The running cost is measured down to a monthly number. The whole thing took two weeks, it was built in the open, and anyone can read the code.

01 — The Tools 6 tools · built in TypeScript

What the six tools do.

Each tool handles one job. The AI is told which jobs exist and picks the right one for your request. Two of the tools run on a faster, cheaper version of the AI, because their work is simple sorting. The other four run on a stronger version, because their work needs real thinking.

Fast model
triage_inbox
Goes through your unread Linear notifications, sorts them by how urgent they are, and says what to do about each one.
Strong model
scope_issue
Takes a new, rough task and turns it into a plan: the steps to do it, what "finished" should look like, the risks, and a rough size estimate.
Fast model
find_orphans
Finds tasks that have gone quiet, gotten stuck, or have nobody assigned to them, and explains why each one needs attention.
Strong model
audit_velocity
Looks at how much work a team finished over recent weeks, and explains whether they are speeding up, slowing down, or just having a normal stretch.
Strong model
compose_update
Writes a Slack or email status update from what the team actually did. Give it a few past updates and it copies the team's writing style.
Strong model
weekly_summary
Writes a short, readable recap of the team's week. A few plain paragraphs instead of a long list of bullet points.
Why two models The faster model is Claude Haiku. The stronger one is Claude Sonnet. Sorting an inbox is quick work, so it uses the cheap fast model. Planning a task needs careful thought, so it uses the stronger one. This keeps the cost down, because the tools you would run every day are the cheap ones.
02 — Testing 30 test cases · automatic scorecard

How I know it works.

An AI gives a slightly different answer every time you ask it something. So "I tried it and it looked fine" does not prove much. I built a test that runs all six tools and checks the answers automatically, then writes out a plain pass-or-fail scorecard with one row per tool. Most people who build these tools skip this step. The gray box below is one of the thirty test cases.

one of 30 test cases [ TEST CASE ] 
{
  tool: "triage_inbox",
  name: "every triaged item is well-formed",
  args: { max_count: 10 },
  checks: [
    expectOk(),
    check("valid priority, summary, action", (o) => {
      for (const it of structured(o).triaged) {
        if (!PRIORITIES.has(it.priority)) throw Error("bad priority");
        if (!it.summary?.trim())          throw Error("empty summary");
      }
    }),
  ],
}
What the test actually does The test does not fake anything. It starts the real tool, the same way the AI assistant would, and gives it real requests. Then it checks each answer against a short list of rules: is the answer shaped correctly, do the numbers add up, and does a bad request get refused with a clear message. It even runs without the paid AI keys, doing whatever checks it can for free. The result is the scorecard, one row per tool.
03 — Security full security review in the repo

What could go wrong, and what I did about it.

This tool acts on a real company's data, so I had to think like someone trying to break it. Three decisions do most of the protecting.

It can only read
None of the six tools can delete or change anything in Linear. They read information and explain it, and that is all. The code to delete a task was never written, so there is no way to trick the AI into deleting one.
The AI never touches the facts
The AI does the thinking, like deciding what is urgent or writing a summary. It never handles the real web links to your tasks. My own code adds those, straight from Linear. So the AI cannot send you to a made-up link, because it never gets to write one.
Sneaky instructions go nowhere
Task descriptions are written by many different people. Someone could hide an instruction inside one, hoping the AI follows it. I cannot make that impossible. But there is nothing harmful for the AI to do here. The worst case is one notification sorted into the wrong pile.
It has no special powers The tool signs in to Linear with my own normal account. It can see and do exactly what I can already see and do, and nothing more. Every time a tool runs, it writes a short note to a log: which tool ran, what was asked, and when. The log leaves out anything private on purpose. This kind of careful access control is the work I did for fifteen years building security systems for banks and a healthcare company.
In review
Things I did on purpose
  • Six focused tools, instead of a copy of every Linear button. Each tool checks its own request before doing anything, so a bad request fails early and with a clear message.
  • Not every job needs heavy AI. Finding stuck tasks is really just a database search. Linear does the searching, and the AI only writes the short explanation. Using AI for the whole thing would be slower and cost more.
  • For each job, the tool asks Linear for everything it needs in one request, instead of dozens of small back-and-forth ones. That makes it faster and cheaper.
  • Two layers of testing. A quick set of checks that runs instantly and free, plus the bigger scorecard test that runs against the live AI.
  • Every tool reports what it spent on AI. That fed an honest cost write-up: roughly $2.60 a month for normal use.
What I would do differently
  • It signs in with my personal Linear access. That is the right call for a tool one person runs on their own computer. A shared, company-wide version would need a full login system, which is a larger job.
  • It spots stuck tasks by looking for a "blocked" label. Linear has a more exact way to record what is blocking what, and a later version should use it.
  • The scorecard test has to be started by hand. Setting it to run on its own after every change is a clear next step.
  • There is a cost-saving feature wired in that is not doing anything yet, because of a size limit on how it works. I left it in and wrote down why, rather than hiding it.
  • It is built for one person. Making it safe for a whole company to share is a separate, bigger piece of work.
