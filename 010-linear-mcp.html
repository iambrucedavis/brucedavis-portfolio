Linear MCP

An MCP server for Linear. Most MCP servers wrap a SaaS API one endpoint at a time and leave the model to sort it out. This one exposes six opinionated tools that match how a senior engineer runs an issue tracker — and it ships with the three things most MCP servers skip: an evaluation harness, a threat model, and a cost analysis.

6 tools MCP server TypeScript Open source
The Brief

Most MCP servers for a SaaS tool are thin wrappers. They expose every API endpoint — get issue, update issue, create comment — and leave the language model to orchestrate REST calls. That works. But it treats the model as an RPC caller, and the user's actual intent — what needs my attention? — is nowhere in the tool list.

This server takes a different bet: six opinionated primitives, each one a whole job. Fetch the right data, apply judgment, return a structured, validated result. The model orchestrates intent, not endpoints. And the tool surface is deliberately read-only — there is no delete_issue. Omission is a design statement and a security boundary at once.

The tools are half the project. The other half is the part most MCP servers skip — an evaluation harness, a threat model, and a cost analysis. Built over fourteen days, in public, in TypeScript. Open source, MIT.

01 — The Tools 6 tools · 2 models · TypeScript + Zod

Six tools, picked on purpose.

Each tool does one job. Inputs and outputs are both validated with Zod. The model is chosen per tool — cheap, fast Haiku for classification, stronger Sonnet for reasoning and writing.

Haiku
triage_inbox
Classify unread Linear notifications by urgency, summarize each in a line, and suggest a concrete next action.
Sonnet
scope_issue
Break a freshly filed issue into subtasks, testable acceptance criteria, risks with mitigations, and an effort estimate.
Haiku
find_orphans
Surface stale, blocked, or unassigned issues — each with a one-line note on why it slipped and what to do.
Sonnet
audit_velocity
Diagnose a team's sprint-velocity trend with reasoning — distinguishing a real signal from cycle-to-cycle noise.
Sonnet
compose_update
Draft a Slack or email status update from a team's recent activity, matched to the team's voice via past examples.
Sonnet
weekly_summary
Write a narrative summary of a team's activity over a date range — the story of the period, not a bullet list.
Model selection Match the model to the cognitive load. Triage is fast pattern-matching, so it runs on Haiku; scoping an issue is genuine planning, so it runs on Sonnet with reasoning on. The cost lever underneath: the cheap tools are also the high-frequency ones — you triage an inbox daily. Frequency times unit cost is what shows up on the bill, which is why the per-tool model choice matters.
02 — The Eval Harness 30 cases · end-to-end · pass/fail report

Most MCP servers can't tell you if they work.

"It worked when I tried it" is not a quality bar for a system that behaves a little differently every run. The harness defines 30 ground-truth cases — five per tool — each an input plus criteria for "good enough."

tests/evals/cases.ts · one of 30 [ GROUND TRUTH ] 
{
  tool: "triage_inbox",
  name: "every triaged item is well-formed",
  args: { max_count: 10 },
  checks: [
    expectOk(),
    check("valid priority, summary, action", (o) => {
      for (const it of structured(o).triaged) {
        if (!PRIORITIES.has(it.priority)) throw Error("bad priority");
        if (!it.summary?.trim())          throw Error("empty summary");
      }
    }),
  ],
}
Test the real thing, not a mock The harness spawns the actual built server as a subprocess and drives it through a real MCP client over stdio — the exact path Claude Desktop uses. So it exercises the JSON-RPC wiring, the input validation, the structured-output contract, all of it. The criteria are structural, not exact-match — the data lives in whoever's Linear workspace runs it. It degrades gracefully too: with no API keys it still runs every deterministic case and reports the rest as skipped. The output is a markdown pass/fail table per tool.
03 — The Security Model threat model · documented in SECURITY.md

A read-only server, on purpose.

An MCP server hands a language model real capabilities. That deserves a real threat model — and most MCP servers don't have one. Three decisions carry most of the weight.

Omission as a control
No destructive tool exists. No delete_issue, no update_issue, no raw query. You cannot misuse a capability that was never built. The worst case for the whole server is a bad read.
Judgment vs. facts
The model supplies judgment — a priority, a summary. It never sees or produces issue URLs; those are joined in from the real Linear response. A hallucinated link is structurally impossible, not just unlikely.
Bounded injection
Issue text is untrusted — many authors. Prompt injection isn't "solved," but a successful one has nowhere to go: nothing destructive to reach, no facts to forge, no secrets in context.
The authorization boundary The server authenticates to Linear with a Personal API key, which means it inherits exactly its operator's Linear permissions — nothing escalatable. The model gets no privilege the user didn't already have. Every tool call is logged — tool, inputs, timestamp — to a stderr audit trail, data-minimized: compose_update logs the count of voice samples, never their contents. Fifteen years of regulated-environment IAM work, showing up where it belongs.
In review
Craft decisions
  • Six opinionated primitives, not a wrapper. Each tool is one file, with a Zod schema on the input and the output — one schema serves the MCP contract, the Anthropic structured-output format, and runtime validation.
  • Not every tool needs heavy reasoning. find_orphans is a database query — Linear's filter engine finds the orphans; the model only writes the summary line. The model is a scalpel, not a hammer.
  • One GraphQL query per call, not the SDK's lazy object graph — avoids an N+1 of one round-trip per notification.
  • Two test layers: 56 offline contract tests that run with no keys, and the 30-case live eval harness. Fast checks and graded checks, kept separate.
  • Token usage is reported on every tool call, which feeds an honest cost analysis — roughly $2.60/month for a normal user, and a clear account of where it goes.
What I'd do differently
  • Personal API key, not OAuth. Correct for a local single-user server — the cleanest authorization boundary there is — but a hosted, multi-tenant version needs OAuth with minimum-scope tokens.
  • find_orphans detects "blocked" with a label-name heuristic. Linear models explicit issue relations; a v2 should use those for precision.
  • The eval harness runs by hand. It isn't wired into CI yet — that needs live API secrets in the CI vault.
  • Prompt caching is wired but inactive — the system prompts sit under the minimum cacheable-prefix size. Left in and documented rather than hidden.
  • Single-user by design. Multi-tenant isolation, per-user audit attribution, and rate limiting are all v2 — the v1 is built so they're additions, not rewrites.
